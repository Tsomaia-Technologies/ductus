# Task: 007-clock-processor

**Global References:** 
- `rfc-001.implementation-guide.md` (MUST READ for Deterministic Timers)
- `rfc-001.revision-06.md` (Section 8.1 The Clock Utility)

## 1. Objective & Responsibility
Implement the `ClockProcessor`. Traditional systems rely heavily on scattered `setTimeout` and `Date.now()`, which inherently breaks deterministic replay testing. The ClockProcessor acts as the single Suprachiasmatic Nucleus for the entire engine. It regulates time by injecting `TICK` events into the Hub. Every other processor relies on the stream of `TICK` events, not the system clock.

## 2. Invariants & Global Contracts (The Happy Path)
- **The Temporal Invariant:** Two runs of the system with the exact same sequence of events, including `TICK` events, MUST produce identical state outcomes.
- **Dual Temporal Modes:** The processor MUST dynamically regulate its output based on the Hub's mode. 
    1. **Live Mode:** Dispatches `TICK` via standard `setInterval` (e.g., every 1000ms).
    2. **Replay Mode (Silent Hydration):** Instead of firing every 1000 real-world milliseconds, the processor scans the replayed events and dispatches `TICK` events *synchronized* with the `timestamp` recorded on the historic events.

## 3. I/O Boundaries & Dependencies
- **No Direct System Clock Access Exception:** This is the *only* processor allowed to query the real `Date.now()` (during Live Mode) or instantiate an active `setInterval`. 
- **The Hub Parameter:** It MUST take the `MultiplexerHub` as a constructor parameter to yield its `TICK` events directly into the spine.

## 4. Exact Event Signatures (Contract Adherence)
- **Listens to:** `SYSTEM_START` (To begin the interval), `SYSTEM_HALT`, or `CIRCUIT_INTERRUPTED` (To destroy the interval).
- **Yields (Durable):** `SYSTEM_TICK` (Event payload must contain the current logical timestamp. Must be `durable-draft`).

## 5. Lifecycle & Unhappy Path
- **Graceful Teardown:** When the engine gracefully terminates (e.g. `SYSTEM_HALT`), or encounters a global panic (`CIRCUIT_INTERRUPTED`), the ClockProcessor MUST explicitly invoke `clearInterval()` and nullify any pending timeouts. A hanging interval will prevent the Node.js process from cleanly exiting.
- **Hydration Sync:** If replaying a gap where the system was offline for 3 days, do NOT yield 250,000 `TICK` events. Fast-forward the logical time to the next recorded event's timestamp.

## 6. Required Execution Constraints
- Define private properties holding the `intervalRef` and `mode`.
- The processor must listen to the incoming stream. When it receives a mode-changing system event, it immediately updates its strategy.

## 7. Definition of Done
1. **The Teardown Proof:** Write a Jest test initializing the Clock in Live Mode. Assert that `SYSTEM_TICK` events begin yielding. Then, yield `SYSTEM_HALT` to the input stream. Assert that the Node instance has zero active timeouts/intervals pending (meaning `clearInterval` was successful).
2. **The Time Travel Proof:** Yield historical events dated January 1st to the Input Stream. Assert that the `SYSTEM_TICK` events generated by the Clock matching those periods have `timestamp` fields matching January 1st, regardless of what `Date.now()` is in the Jest environment.
