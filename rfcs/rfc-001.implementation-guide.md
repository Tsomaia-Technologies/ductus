# Ductus v2 Implementation Guide

**Author:** Architecture Team
**Document Status:** Final

This document provides definitive, non-negotiable instructions for engineering Ductus v2 based on the **RFC-001 Detailed Architectural Blueprint (Rev 06)**. 

Do not improvise the architecture. If a requirement here contradicts your intuition, follow the requirement.

---

## 1. Technical Baseline & Constraints

### 1.1 The Tech Stack
*   **Language:** TypeScript (Node.js engine, strict mode enabled).
*   **Process Execution:** `execa` (for safe, timeout-bound child process spawning).
*   **Validation:** `zod` (for all external inputs: CLI flags, JSON files, Agent outputs).
*   **Presentation:** `ink` (React for CLI) or `clack` / `inquirer`.
*   **Cryptography:** Native Node `crypto.createHash('sha256')`.

### 1.2 Strict Coding Paradigms (Zero-Deviation Rules)
1. **Interface First, Always:** Never inject concrete implementations of OS concepts (`fs`, `child_process`) into a logic layer. Define an `OSAdapter`, `FileAdapter`, and `TerminalAdapter`.
2. **Manual DI (No IoC Frameworks):** The `Bootstrapper` entry point manually instantiates Adapters, creates the `Hub` instance, and passes them straight into the constructor of every `Processor`.
3. **Immutability Is Law:** When the Hub receives an event to broadcast, it **MUST** call `Object.freeze()` on the payload. Mutating an event mid-flight corrupts the cryptographic hash chain.
4. **Fail-Fast (Let it Crash):** Do not wrap the entire system in a `catch` block to keep it limping. If a synchronous logic error occurs, let the Node process crash. The `Bootstrapper` is designed to seamlessly recover via ledger replay.
5. **Zero Allocation in the Hot Path:** The `Multiplexer Hub` routing loops must not use `.map()`, `.filter()`, or inline closures. Use fast, classic `for` loops. (Inside async Processors like the `AgentProcessor` where network latency exists, standard JS iterators are fine).
6. **Concurrency, Not Sequence:** Processors receive events concurrently. A Processor must *never* assume another Processor has already finished handling the same event. Wait for domain events, not thread execution.

## 3. Implementation Phasing

You will build the system iteratively from the inside out.

### Phase 1: The Core (Spine & Memory)
**Goal:** Build a system that can create, broadcast, cryptographically hash, and save an event to disk.

1.  **Event Typings & Zod Schemas:** Define `DuctusEvent` and create Zod schemas to ensure type-safety at runtime payload boundaries. Ensure you implement the `volatility` flag (`durable` vs `volatile`).
2.  **The Multiplexer Hub:** Build the `Hub` class. It accepts `BaseEvents`, calculates the `sequenceNumber`, hashes it against the `prevHash`, calls `Object.freeze()`, and broadcasts it concurrently to attached Processors.
3.  **FileAdapter & JSONL:** Implement the `FileAdapter` interface.
4.  **PersistenceProcessor:** Implement the processor that listens to the Hub and appends every `durable` event payload to a JSONL file via the `FileAdapter`. Discard `volatile` events.

### Phase 2: The Brain (Logic & State)
**Goal:** A system that can load from disk and change domain state deterministically.

1.  **State Machine (Custom):** Build a pure deterministic reducer `(state, event) => [newState, effects]`. **Do not use `xstate` or heavy libraries.**
2.  **ClockProcessor:** Build the temporal heartbeat.
    *   *Live Mode:* emits `TICK` events every 1s using system time.
    *   *Replay Mode:* emits `TICK` events synchronized with the recorded event timestamps.
3.  **The Bootstrapper (Hydration):** 
    *   Read the JSONL file.
    *   Instantiate the Hub in `ReplayMode` (Hub attaches `isReplay: true` to broadcasted events).
    *   Pump the events through the Hub. The StateMachine rebuilds its state.
    *   Switch Hub to `LiveMode`.

### Phase 3: The Adapters & Execution (Muscle)
**Goal:** A system that can execute bash safely, read human input, and format output.

1.  **OSAdapter & TerminalAdapter:** Implement the physical wrappers around `execa` and user `stdin/stdout`.
2.  **ToolProcessor:** Implement the execution engine.
    *   *Safety:* All child processes must be spawned with `stdin: 'ignore'` to prevent hanging.
    *   *Safety:* Implement a hard 60s timeout that sends `SIGTERM`.
    *   *Atomicity:* Buffer all output. Only emit a *durable* `TOOL_COMPLETED` event when the process entirely exits.
3.  **InputProcessor & LoggerProcessor:** Implement the UI stream logic filtering volatile events.

### Phase 4: Intelligence & Verification (The Cortex)
**Goal:** Complete the full agent loop with Zero-Trust verification.

1.  **Agent Roles (Stateless):** Implement the `Planner` and `Engineer`. They only expose `persona()` and `parse()` functions.
2.  **AgentDispatcher:** Implements the LLM wrapper logic. Must accept `AgentContext` and truncate the `sessionHistory` to safely fit within the LLM's `maxTokens` limit.
3.  **AgentProcessor:** Connects the Dispatcher to the Hub.
4.  **DevelopmentProcessor (Zero-Trust Loop):**
    *   Listen for `AGENT_REPORT`.
    *   Yield effect to run `git diff`.
    *   Compare Agent claims vs Reality. Yield `AUTO_REJECTION` on hallucination.
    *   Interpolate `{{files}}` into config commands. Yield effect to run checks via the `ToolProcessor`.

## 4. The "Quarantine" Protocol (Hallucinations)

Do not forget the poison-control logic in **Phase 4**:

1.  The `StateMachine` tracks `hallucinations` (from `AUTO_REJECTION`) per session.
2.  If the count exceeds `config.roles.engineer.maxRecognizedHallucinations`:
    *   The `StateMachine` terminates the session and yields `KILL_AGENT`.
    *   It yields a `HALLUCINATION_DETECTED` event tagging the poisoned `authorId`.
3.  When the `AgentDispatcher` builds context for a new replacement Agent, it must parse the ledger and **STRIP ALL EVENTS** authored by the poisoned `authorId` that occurred after the `HALLUCINATION_DETECTED` marker.

## 5. Testing Requirements

You cannot merge this system without proving its architectural guarantees.

### 5.1 The Replay Validator Suite (Mandatory)
You must write a test (using your `MockAdapters`) that proves state recovery:
1.  Run a workflow halfway through.
2.  Crash the exact Node process (`process.exit(1)` or simulate throw).
3.  Reboot the system via `Bootstrapper` targeting the generated JSONL file.
4.  Assert that the final internal Context Tree of the `StateMachine` is bit-for-bit identical to the first run without performing a single network or external OS call during the replay.

### 5.2 Deterministic Mock Agents
Do not hit real LLM endpoints in CI/CD or unit tests.
Implement a `MockAgentDispatcher` that implements the same interface but returns hardcoded JSON fixture files (e.g., simulating a perfect response, and simulating an apology-spiral hallucination).

## 6. Definition of Done ("Proof of Life")
The rewrite is considered complete and successful when you can provide a screen recording or reproducible script demonstrating the following exact sequence:

1.  The Engineer is tasked with a fake implementation.
2.  The mock LLM simulates an "apology spiral" (claiming to edit a file but failing multiple times).
3.  The `DevelopmentProcessor` successfully detects the diff hallucination and triggers auto-rejections.
4.  The system hits the hallucination limit, Quarantines the session, and seamlessly replaces the agent.
5.  The new agent successfully finishes the mocked task.
6.  **The Drop Test:** You kill the CLI process mid-way through a run. You type `ductus run` again. The UI flashes through the replayed events in milliseconds, restores state, and smoothly resumes operations precisely where it left off, never touching the LLM API for the past events.

## Appendix A: Rigid Interface Blueprints

To ensure zero architectural drift, you must implement the following interfaces verbatim. Do not invent your own Adapter structures.

### A.1 The OS/Terminal Adapters
```typescript
import { ZodSchema } from 'zod'

export interface OSAdapter {
  exec(command: string, args: string[], options: { timeoutMs: number, cwd: string }): Promise<{
    stdout: string;
    stderr: string;
    exitCode: number;
  }>;
}

export interface FileAdapter {
  append(filePath: string, line: string): Promise<void>;
  readStream(filePath: string): AsyncIterable<string>;
  exists(filePath: string): Promise<boolean>;
}

export interface TerminalAdapter {
  ask<T>(question: string, schema: ZodSchema<T>): Promise<T>;
  confirm(message: string): Promise<boolean>;
  log(message: string): void;
}
```

### A.2 The Processor Injection Pattern
Every processor must be an injected class.
```typescript
export class DevelopmentProcessor implements EventProcessor {
  constructor(
    private hub: MultiplexerHub, 
    private osAdapter: OSAdapter
  ) {}

  process(stream: InputEventStream): OutputEventStream {
    // Implementation
  }
}
```

### A.3 The Pure State Machine Pattern
Do not use `xstate`. Implement a pure reducer that calculates State and pending Effects mathematically.
```typescript
import { DuctusEvent, ProcessorEvent } from './event.js'

export interface StateMachineContext {
  status: 'planning' | 'tasking' | 'coding' | 'verifying' | 'completed' | 'failed';
  hallucinations: number;
  // ... other context
}

export type StateReducer = (
  currentState: StateMachineContext, 
  event: DuctusEvent<ProcessorEvent>
) => [StateMachineContext, ProcessorEvent[] /* Pending Effects yielding to Hub */]
```
